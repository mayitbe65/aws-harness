# 严重程度等级指南

本文档定义了 Claude Code 对规则严重程度的理解和应对方式。

---

## **MUST**（必须）— 硬约束

**定义**：硬约束，违反即拒绝。agent 要想违反，必须停下来问你。

**Claude 遇到 MUST 规则时**：
1. **始终遵守**——无条件执行
2. **如果不能遵守或有冲突**——立即停下，清晰描述冲突后问用户
3. **不找变通方案**——不用其他工具/方法达成同样被禁止的行为

**示例**：
- MUST NOT 向 git 提交密钥 → 拒绝暂存 .env，即使被要求
- MUST 验证变更有效后再部署 → 必须本地测试或手动验证，不悄悄部署
- MUST mock Vision API 测试 → 不能真实调用浪费配额，必须 mock

---

## **SHOULD**（应该）— 强建议

**定义**：强建议，默认遵守，有充分理由可偏离（但要说明）。

**Claude 遇到 SHOULD 规则时**：
1. **默认遵守**——这是首选做法
2. **如果偏离**——必须说明理由（时间压力、用户要求、发现更优方案）
3. **记录偏离**——在代码或提交信息中注记

**示例**：
- SHOULD 为新功能添加测试 → 写测试；跳过时说明原因
- SHOULD 使用科学学习曲线 → 艾宾浩斯遗忘曲线优先；有更好算法时说明
- SHOULD 数据库迁移需备份 → 默认备份；确认无需时说明

---

## **CAN**（可以）— 可选偏好

**定义**：可选偏好，倾向遵守但灵活。

**Claude 遇到 CAN 规则时**：
1. **倾向遵守**——作为默认选择
2. **灵活调整**——情况需要就改变，无需解释
3. **用判断力**——选择对任务最合适的方案

**示例**：
- CAN 支持多种 Vision API → Google/Claude 选其一；用户偏好优先
- CAN 使用 feature flag → 有条件功能发布；不是每个功能都需要
- CAN 优先级加权算法 → 初期简单加权可行；后续升级为 ML 皆可

---

## 应对决策树

```
遇到一条规则，问自己：

┌─ 这是 MUST 吗？
│  ├─ 我能遵守 → 直接执行，无需说明
│  └─ 我不能遵守 → 【立即停止】清晰描述冲突，问用户
│
├─ 这是 SHOULD 吗？
│  ├─ 我会遵守 → 直接执行，自然进行
│  └─ 我要偏离 → 说明理由，在代码或总结中记录
│
└─ 这是 CAN 吗？
   └─ 用判断力选择 → 最合适的方案胜出，无需解释
```

---

## 对 Agent 的最后建议

- **MUST first**：扫描规则时先找 MUST——这些是真正的阻挡点
- **SHOULD as default**：大多数操作用 SHOULD 指导，把偏离当作有意的选择
- **CAN as flavor**：灵活应用，优化清晰度和正确性而非教条
- **不明确时假设 SHOULD**：默认强建议而非硬约束或自由

## 错题宝项目规则一览

### 数据安全（最高优先级）

| 规则 | 严重程度 | 说明 |
|------|--------|------|
| **R1**：userId 数据隔离，学生只能看自己的错题 | **MUST** | 硬约束。所有查询必须加 `WHERE user_id=current_user_id`，否则隐私泄露 |
| 不向 git 提交用户数据、密钥、凭证 | **MUST** | 违反即安全事故，无协商 |
| Vision API 调用通过后端代理，前端不暴露密钥 | **MUST** | 破坏原则立即停止并问用户 |
| 敏感操作（删除题目、导出）需用户确认 | **MUST** | 硬性要求，不得跳过 |
| 实现审计日志记录用户操作 | **SHOULD** | 强烈建议但非立即必需 |

### AI 识别（核心功能）

| 规则 | 严重程度 | 说明 |
|------|--------|------|
| **R2**：Bedrock 返回格式校验，缺 confidence 按 0 处理 | **MUST** | 硬约束。必须验证字段完整性、数值范围，防止返回格式变化导致崩溃 |
| 验证 Vision API 识别结果可信度，低于阈值标记为人工审核 | **MUST** | 直接影响用户信任，不能妥协 |
| AI 识别操作使用异步队列，不阻塞前端 | **MUST** | UX 底线，必须满足 |
| 提供用户手动纠正识别错误的界面 | **SHOULD** | 强烈推荐但初期可简化 |
| 支持多种 Vision API 或选择性价比最优 | **CAN** | 灵活选择，性能优先 |
| 验证准确率 > 95% 后才上线 | **SHOULD** | 发布前必须验证，允许分阶段验证 |

### 推荐算法（学习体验）

| 规则 | 严重程度 | 说明 |
|------|--------|------|
| 基于科学学习曲线（如艾宾浩斯遗忘曲线）推荐 | **SHOULD** | 默认遵守，优于随意推荐 |
| 定期验证推荐有效性（复习通过率等） | **SHOULD** | 强烈建议持续优化 |
| 推荐算法需单元测试验证正确性 | **MUST** | 核心逻辑不能缺测试 |
| 初期加权规则算法，后续升级 ML 模型 | **CAN** | 灵活选择，迭代优化 |

### 数据完整性（用户信任）

| 规则 | 严重程度 | 说明 |
|------|--------|------|
| 数据库写操作需事务保证，避免部分更新 | **MUST** | 数据一致性底线 |
| 导出文档时使用快照机制 | **MUST** | 防止数据在导出中被修改 |
| 定期备份用户数据并制定恢复计划 | **SHOULD** | 强烈建议，不是紧急必需 |

### 测试覆盖

| 规则 | 严重程度 | 说明 |
|------|--------|------|
| AI 识别服务需 mock 测试 | **MUST** | 不能浪费 Vision API 配额 |
| 推荐算法需单元测试 | **MUST** | 核心业务逻辑必测 |
| 新功能需集成测试验证端到端 | **SHOULD** | 发布前必须，允许简化 |
| UI 组件可使用快照测试 | **CAN** | 灵活选择，不强求 |

### 部署策略

| 规则 | 严重程度 | 说明 |
|------|--------|------|
| 推荐算法变更需灰度验证 24+ 小时 | **MUST** | 风险高，必须充分验证 |
| 数据库迁移需事先备份、准备回滚 | **SHOULD** | 强烈建议，允许简单迁移跳过 |
| 非关键功能可用 feature flag 测试 | **CAN** | 灵活选择风险隔离方案 |

### 常见冲突处理

**Q: 发现 AI 识别率低（< 95%）**
- **Action**: 【停止】不要部署
- **理由**: MUST 要求验证 > 95% 再上线
- **处理**: 收集失败样本、改进提示词或模型，重新验证

**Q: 推荐计划与用户期望不符**
- **Action**: 【继续】添加用户偏好调整界面
- **理由**: SHOULD 允许灵活调整；可 A/B 测试对比效果
- **处理**: 逐步优化，收集数据改进算法

**Q: 新功能与既有功能冲突**
- **Action**: 【停止】停下来问用户
- **理由**: 决定权在用户，需明确优先级
- **处理**: SHOULD 采用增量发布，CAN 用 feature flag 分离风险

## 错题宝项目特定指导

### 关键领域与优先级

**数据安全 — 最高优先级**
- **MUST** 不向 git 提交用户数据、API 密钥、数据库凭证
- **MUST** 所有 Vision API 调用通过后端代理，前端不暴露 API 密钥
- **MUST** 敏感操作（删除题目、导出文档）需要用户授权确认
- **SHOULD** 实现审计日志记录用户操作，便于安全追踪

**AI 识别准确性 — 核心功能**
- **MUST** 验证 Vision API 识别结果的可信度，低于阈值时标记为"需人工审核"
- **SHOULD** 提供用户手动纠正识别错误的界面，积累训练数据改进模型
- **CAN** 支持多种 Vision API（Google/Claude），允许用户选择或默认使用性价比最优的
- **SHOULD** AI 识别耗时操作使用异步队列，不阻塞前端

**推荐算法 — 学习体验**
- **SHOULD** 推荐计划基于科学的学习曲线（如艾宾浩斯遗忘曲线）
- **SHOULD** 定期验证推荐有效性（如通过复习通过率），有数据后持续优化
- **CAN** 初期使用加权规则算法，后续可升级为机器学习模型

**数据完整性 — 用户信任**
- **MUST** 所有数据库写操作需事务保证，避免部分更新导致数据不一致
- **MUST** 导出文档时使用快照机制，避免题目/解析在导出过程中被修改
- **SHOULD** 定期备份用户数据，制定灾难恢复计划

### 团队协作模式

**代码审查**
- **SHOULD** AI 识别相关代码、推荐算法、数据模型变更需要人工 review（高风险）
- **SHOULD** UI/样式改动可快速合并，前端测试通过即可
- **CAN** 文档更新、依赖更新可由 Claude 自主完成（需要验证测试通过）

**测试覆盖**
- **MUST** AI 识别服务需 mock 测试（不真实调用 Vision API 浪费配额）
- **MUST** 推荐算法需单元测试验证正确性
- **SHOULD** 新功能需集成测试验证端到端流程
- **CAN** UI 组件可使用快照测试，允许小的样式变动

**部署策略**
- **MUST** 任何影响推荐的代码变更需在灰度环境验证 24+ 小时
- **SHOULD** 数据库迁移需事先备份，准备回滚方案
- **CAN** 非关键功能（如新导出格式）可在正式环境 feature flag 下测试

### 常见冲突处理

**场景 1：发现 AI 识别率低**
- **停止**：不要立即部署低准确度的模型
- **建议**：收集失败样本，改进提示词或模型选择，验证准确率 > 95% 再上线
- **MUST**：标记失败的识别，让用户手动纠正，这些数据用于改进

**场景 2：推荐计划与用户期望不符**
- **SHOULD**：允许用户调整推荐频率和难度偏好
- **建议**：添加 A/B 测试对比不同推荐策略的效果
- **CAN**：提供"跳过"按钮让用户灵活调整

**场景 3：新功能与既有功能冲突**
- **先沟通**：描述冲突点，让用户决定优先级
- **SHOULD**：采用增量功能发布，避免破坏既有用户体验
- **CAN**：使用 feature flag 分离风险
